# AI・社会兆しキュレーションアプリ「きづき」要件定義書

版数: v1.0（ドラフト）
作成日: 2026-07-14
対象読者: 開発者本人（個人利用）

---

## 1. 概要・目的

社会とAIに関する「兆し（トレンドの初期シグナル）」を、複数分野にわたって自動収集・要約し、毎日キャッチできるようにするニュースキュレーションアプリ。単なる新着順の羅列ではなく、**注目度・トレンド性の高い記事を優先的に提示**し、かつ**ユーザーの好みに徐々に適応しながらも多様性を失わない**設計とする。

既存の個人プロジェクト `ai-tech-catchup`（本リポジトリ同階層）が「収集→要約→分類→一覧表示」の基本パイプラインを持っているため、「きづき」はそのパイプラインを土台にしつつ、以下を新規に追加した発展形として位置づける。

- カテゴリ体系を5分野に再編
- トレンド・注目度スコアリング
- フィードバックによる段階的パーソナライズ（EMA）
- GA的突然変異による多様性維持
- ブックマーク機能
- ダークモード
- ホーム画面に追加できるPWA（後述1.2節）

### 1.1 実現したい機能・解決したい課題（背景ヒアリング）

社会科学系の研究室（社会課題をデータサイエンス・シミュレーション・ゲーミング等の技術で解決することに取り組む研究室）での利用を想定している。こうした研究室では、日々の知識のアップデートと社会の仕組みの理解が欠かせず、扱う情報の幅は「細かい日常的な疑問」から「国の制度的な話」「AIサービス・動向のキャッチアップ」まで広い。

そこで、注目されている情報が自動で流れてくるフィード型のアプリを作り、研究室メンバーが毎日効率的に情報収集できるようにすることを目的とする。情報収集の要件として、**収集元を幅広く保ち、特定の媒体・分野に偏らないこと**を重視する（4.1節・8章参照）。

### 1.2 プラットフォームの方針

本来はネイティブのiOSアプリとしての提供が理想だが、Apple App Storeの審査・申請手続きが煩雑なため、v1では**Webアプリ（PWA）として実装し、スマートフォンのホーム画面に追加して起動できること**を目標とする。具体的には以下を満たす。

- `manifest.json` によるアプリ名・アイコン・テーマカラーの定義
- ホーム画面追加時にブラウザのアドレスバー等が表示されない「standalone」表示
- Service Worker によるオフライン時の簡易フォールバック表示

## 2. 対象ユーザー

社会科学系研究室のメンバー（研究者・大学院生等）を主な利用者として想定する。当面は少人数での利用のため、認証・マルチユーザー対応は不要（個人／少人数の共用ツールという位置づけ）。

## 3. 対象カテゴリ（5分野）

- AI
- 制度
- 社会×データ
- 学術
- 新事業

各記事は1つ以上のカテゴリタグ・ハッシュタグ（例: `#AIエージェント`、`#国内動向`）を持つ。

## 4. 機能要件

### 4.1 ニュース収集（F-01）

- 各カテゴリについて、日次で新着ニュース・記事を収集する。
- 収集元: note、PR TIMES、その他ニュースメディア・企業プレスリリース、Qiita・ブログなど複数ソース。
- **収集元は特定の媒体・分野に偏らないよう、幅広く保つ**（1.1節参照）。具体的には「テック専門メディア」「政府・公的機関」「学術(arXiv)」「個人/実践者の発信」「プレスリリース」の異なる性質のソースを組み合わせる構成とし、特定カテゴリ・特定媒体への偏りが生じていないか `/sources` 画面や4.3節のトレンドスコアの `source_count`（複数媒体掲載スコア）で継続的に確認できるようにする。
- 出典元名を表示し、外部リンクで元記事に遷移できるようにする。
- 選定対象は**公開から1週間以内**、かつ新規性・新しい情報を含むもの。
- 1回の収集で新規に取り込む記事数は**最大15件**とする（LLM要約の無料枠を使い切らないための制限。`MAX_NEW_ARTICLES_PER_RUN`で調整可能）。

### 4.2 AI要約（F-02）

- タイトル（1〜2行）
- 要約文（3〜4文、平易な日本語）
- ハッシュタグ（2〜3個、自動抽出）
- 掲載日の相対表示（例:「2日前」）

出力はJSON形式（`{title, summary, tags[], category[], published_at}`）に固定し、Function Calling/Structured Outputで生成する。既存アプリの `lib/summarize.js` の方式を踏襲する。

### 4.3 トレンド・注目度スコアリング（F-03）

各カテゴリ内で当日の「兆し」として注目度が高い記事を優先表示する。

**スコアリング式（v1案）**

```
trend_score = w1 * 言及頻度スコア
            + w2 * 複数媒体掲載スコア
            + w3 * 鮮度スコア
            + w4 * SNS言及スコア（取得できる場合のみ）
```

- 言及頻度スコア: 同一トピック（タイトル・要約の類似度クラスタリング、または共通キーワード）が何件収集されたか。
- 複数媒体掲載スコア: 同一トピックが異なる出典元で何件確認できたか。
- 鮮度スコア: 公開からの経過時間が短いほど高スコア（指数減衰）。
- SNS言及スコア: v1では取得手段が未確定のため0固定でよい（10章参照）。将来的にX API等を追加。
- 重み `w1〜w4` は設定ファイルで調整可能とする（初期値: w1=0.4, w2=0.3, w3=0.3, w4=0）。

トップ画面には「最終更新: YYYY/MM/DD・N件の兆し」を表示する。

### 4.4 フィードバックによるパーソナライズ（段階的学習）（F-04）

- 各記事に「👍 もっと見たい」「👎 あまり興味ない」ボタンを設置。
- カテゴリ・タグ単位でユーザーの好みスコア `preference_score`（初期値0、範囲 -1.0〜+1.0）を保持。
- フィードバックのたびにEMAで更新する。

```
preference_score_new = preference_score_old * (1 - α) + feedback_value * α
```

- `feedback_value`: 👍 = +1.0、👎 = -1.0
- `α`（学習率）: 初期値 0.2（調整可能パラメータ）
- 表示確率の重み付けに反映（完全排除ではなく段階的減衰）。表示スコアは `trend_score * (1 + β * preference_score)` のように合成する（`β` 初期値0.5）。

### 4.5 多様性維持（GA的突然変異メカニズム）（F-05）

- 一定確率 `mutation_rate`（初期値 **7%**、5〜10%の範囲でパラメータ化）で、普段は選ばれにくい/好みスコアが低いジャンルの記事をあえて混入させる。
- 実装方針: 各カテゴリのフィード生成時、`mutation_rate` の確率で「好みスコア下位20%かつtrend_score上位」の記事枠を1件確保する。
- 設定画面または設定ファイルで調整可能とする。

### 4.6 ブックマーク（F-06）

- 記事単位でブックマーク保存（DBに `bookmarked_at` を保持）。
- 画面右上のブックマークアイコンから一覧表示画面に遷移できる。

### 4.7 UI（F-07）

- カテゴリタブでの切り替え（横スクロール、5分野）。
- カード形式のフィード（タイトル・要約・タグ・出典・👍👎・ブックマークボタン）。
- ダークモード切り替え（デフォルトはライトモード、設定はlocalStorageに保持）。
- 最終更新日時・記事件数の表示。
- スマートフォンのホーム画面に追加できるPWA対応（1.2節）。`manifest.json`・アイコン一式・Service Workerを実装済み。

### 4.8 プラットフォーム（PWA）（F-08）

- Apple App Store等ネイティブアプリの審査手続きを避けるため、Webアプリとして実装し、スマートフォンのホーム画面から起動できることをv1のゴールとする。
- `display: standalone` により、ホーム画面から開いた際はブラウザのアドレスバー等を非表示にする。
- Service Workerでネットワーク優先・オフライン時キャッシュフォールバックの簡易対応を行う（本格的なオフライン編集等は対象外）。

## 5. データ設計（既存 `articles` テーブル/`sources` テーブルへの追加案）

### 5.1 `articles` への追加カラム

| カラム名 | 型 | 説明 |
|---|---|---|
| trend_score | float | 4.3節のスコアリング結果 |
| mention_count | int | 同一トピックの言及件数 |
| source_count | int | 掲載媒体数 |
| bookmarked_at | datetime/null | ブックマーク日時（nullなら未登録） |

### 5.2 新規テーブル `preferences`

| カラム名 | 型 | 説明 |
|---|---|---|
| key | text | `category:AI` や `tag:AIエージェント` のようなキー |
| score | float | EMAで更新される好みスコア（-1.0〜+1.0） |
| updated_at | datetime | 最終更新日時 |

### 5.3 新規テーブル `feedback_log`（監査用、任意）

| カラム名 | 型 | 説明 |
|---|---|---|
| id | 連番 | 主キー |
| article_id | FK | 対象記事 |
| feedback_value | int | +1/-1 |
| created_at | datetime | フィードバック日時 |

## 6. 非機能要件

- 個人利用前提のシンプルな構成（大規模なマルチユーザー基盤は不要）。
- 日次バッチでの記事収集・要約・スコアリング更新を想定（既存の `scripts/run-batch.js` 相当の仕組みを拡張）。
- 好みスコア・ブックマークは無期限保存（ストレージ増大時にアーカイブ方針を再検討）。

## 7. 技術構成（既存資産の再利用を前提とした推奨案）

既存 `ai-tech-catchup` が Next.js（App Router）+ SQLite + Claude API + Render（Cron）で構築済みのため、「きづき」も同一スタックを踏襲することを推奨する。

- フロントエンド/バックエンド: Next.js（App Router）、既存の `app/`, `components/`, `lib/` を拡張
- DB: SQLite（既存 `lib/db.js` を拡張し、`preferences`・`feedback_log` テーブルを追加）
- バッチ実行: 既存の Render Cron Job（`ai-tech-catchup-daily-batch`）と同様の構成、または個人PC上のローカル実行でも可
- LLM: 既存同様 Claude API（`@anthropic-ai/sdk`）

新規アプリとして別リポジトリ/別ディレクトリを切るか、既存 `ai-tech-catchup` を拡張するかは10章の未確定事項とする。
