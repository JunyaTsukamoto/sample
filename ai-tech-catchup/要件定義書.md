# 個人用AI技術動向キャッチアップアプリ 要件定義書

版数: v1.0（ドラフト）
作成日: 2026-07-12
作成者: システムアナリスト（本ドキュメントはClaudeによる作成）
対象読者: 開発者本人（研究者・大学院生）

---

## 1. 概要

### 1.1 目的

研究室に所属する研究者・大学院生が、AI・技術分野の最新動向を毎朝効率的にキャッチアップするための、個人専用Webアプリを構築する。単なるニュース集約ではなく、研究室で実際に試せる・模倣できる実践的な情報（ツール活用法、DIY事例、生成AI活用事例など）と、俯瞰的に押さえておくべき業界動向の両方を、区別した形で提示することを目的とする。

### 1.2 利用者

- 本人のみが利用する個人用ツール。認証・アカウント管理機能は不要。

### 1.3 利用シーン

- 毎朝の定点チェック（出勤・研究開始前に5〜10分程度で当日分の情報を確認する想定）。
- 気になった記事は元ソースへ遷移して詳細を読む。

### 1.4 情報の2つの性質（本要件定義の前提）

本アプリが扱う情報には性質が異なる2系統があり、情報源・要約観点・UI上の見せ方をすべて区別して設計する。

| 系統 | 内容 | 想定情報源 | 要約の観点 |
|---|---|---|---|
| ①動向・トレンド系 | 新モデル発表、企業動向、資金調達、規制・倫理など、俯瞰的に把握しておきたい情報 | 大手テックメディア、公式ブログ、プレスリリース、規制当局・学会の発表 | 「何が起きたか」「業界にとっての意味」を簡潔に把握できる要約 |
| ②実践・模倣系 | 研究室で試せる具体的な活用事例・Tips・How-to（ツール活用法、DIY事例、生成AIコンテンツ制作事例など） | Qiita、Zenn、個人ブログ、Hacker News等の実践者発信 | 「何ができるか」「どう真似できるか」「必要な前提条件・コスト」を中心にした要約 |

このほか、専門領域（防災シミュレーション、ABM、GIS）と技術的に交差する情報を横断タグとして扱う（③専門交差系）。

## 2. スコープ

### 2.1 v1に含む機能

- RSS/APIによる記事収集（複数ソース、定期バッチ実行）
- LLMによる要約・カテゴリ分類・タグ付け（動向系／実践系で要約観点を切り替え）
- 記事一覧表示（カード/リスト形式、新しい順）
- カテゴリ・タグによるフィルタリング
- 記事詳細表示および元記事へのリンク
- 実践可能性フラグ（「研究室で試せる」ことが分かる目印）の表示
- 手動再収集（バッチの手動トリガー）

### 2.2 v1に含まない機能（将来検討）

- マルチユーザー対応・認証・権限管理
- 既読/未読管理、お気に入り、ブックマーク機能
- 記事への個人メモ・タグの手動追加編集
- 通知（メール・Slack等へのプッシュ通知）
- 全文検索・横断検索エンジン
- モバイルアプリ化・PWA対応
- 複数言語対応（UIは日本語のみ、記事は英語含む）
- 記事の重複検知の高度化（現状は簡易な重複排除のみ）
- ユーザーフィードバックに基づく推薦・パーソナライズ機能

### 2.3 スコープ境界の考え方

v1は「収集→要約→分類→一覧表示→フィルタ」という基本パイプラインを確実に動かすことを優先し、パーソナライズや通知などの付加機能はv2以降とする。

## 3. 機能要件

| ID | 機能 | 概要 |
|---|---|---|
| F-01 | 記事収集 | 登録済みの情報源（RSS/API）から定期的に新着記事を取得する |
| F-02 | 重複排除 | URL・タイトルの類似度等により明らかな重複記事を除外する |
| F-03 | 記事本文取得 | RSSに本文が含まれない場合、元記事ページから本文相当のテキストを取得する（可能な範囲で） |
| F-04 | 系統判定 | 情報源の設定に基づき、記事を「動向系」「実践系」「専門交差系（タグ）」に振り分ける |
| F-05 | AI要約 | LLMにより、系統に応じた観点で要約文を生成する（詳細は5章） |
| F-06 | AIカテゴリ分類 | 7カテゴリ（4章参照）のいずれか（複数可）に分類する |
| F-07 | 実践可能性フラグ付与 | 実践系記事に対し「試せる度合い」を示すフラグ・スコアを付与する |
| F-08 | 記事一覧表示 | 収集・要約済み記事を新着順に一覧表示する |
| F-09 | カテゴリ/タグフィルタ | カテゴリ・タグ・系統（動向系/実践系）で記事を絞り込む |
| F-10 | 記事詳細表示 | 要約全文、カテゴリ、タグ、元記事リンク、収集日時等を表示する |
| F-11 | バッチ処理 | 収集・要約・分類処理を定期実行する（実行頻度は7章参照） |
| F-12 | 手動再実行 | バッチ処理を手動でトリガーできる（障害時・即時確認したい場合用） |
| F-13 | 処理ステータス確認 | 直近のバッチ実行結果（成功/失敗、件数）を確認できる |

## 4. 情報源の設計

### 4.1 情報源の3系統

| 系統 | 想定ジャンル | 収集方法（案） | 備考 |
|---|---|---|---|
| ①動向・トレンド系 | 新モデル発表、企業・資金調達動向、規制・倫理、研究活動でのAI利用ルール | 大手メディアのRSS、企業公式ブログRSS、arXiv等のAPI | 情報の信頼性・一次情報性を重視 |
| ②実践・模倣系 | ツール活用法、DIY（ラズパイ・3Dプリンタ・IoT）、生成AIコンテンツ制作事例 | Qiita/Zenn API、個人ブログRSS、Hacker News API（Algolia HN Search API等） | 実践者の一次発信を重視。ノイズ（質の低い記事）が混ざりやすい点に留意 |
| ③専門交差系（横断タグ） | 防災シミュレーション・ABM・GISとAI技術の交差点 | ①②の収集結果に対しキーワード/LLM判定でタグ付け。専門メディア（GIS系、防災系）のRSSも追加候補 | 独立した収集パイプラインではなく、①②の記事に付与する横断タグとして扱う |

### 4.2 収集方式の具体案（未確定、選択肢を提示）

1. **RSS中心方式**: 各情報源が提供するRSS/Atomフィードを定期ポーリング。実装が簡単、追加・削除が容易。ただしRSSを提供しないサイトは対象外。
2. **API中心方式**: Qiita API、Zenn（非公式API/RSS併用）、Hacker News Algolia API、arXiv APIなど、各サービスのAPIを個別実装。取得情報の粒度が上がるが実装コストが増える。
3. **ハイブリッド方式（推奨）**: 基本はRSSで広く浅く収集し、Qiita/Zenn/Hacker Newsなど実践系で重要なソースのみAPIで深く取得する。

### 4.3 情報源リストの確定について

具体的な情報源名（例：どの企業ブログ、どのメディア、Perplexity Blog、OpenAI Blog、Google AI Blog、Hugging Face Blog等）は本要件定義の範囲では確定せず、初期セットを別紙（情報源マスタ）として管理し、随時追加・削除できる設計とする（8.3節、10章参照）。

## 5. AI要約・分類のプロンプト設計方針

### 5.1 基本方針

LLM単体に「ニュースを生成させる」のではなく、実際に収集した記事本文（またはRSSのdescription/summary）を入力として与え、その内容の範囲内で要約・分類させる（ハルシネーション対策）。プロンプトには「与えられた本文に基づいてのみ要約すること。本文に無い情報を推測して補わないこと」という制約を明示する。

### 5.2 系統別の要約観点

| 系統 | 要約に含める観点 | プロンプト設計のポイント |
|---|---|---|
| 動向・トレンド系 | ①何が発表・発生したか ②誰が関わっているか（企業・研究機関） ③業界的な意味合い・影響 ④信頼度（一次情報か伝聞か） | 「事実」と「予測・憶測」を区別して記述するよう指示する |
| 実践・模倣系 | ①何が実現できるか ②具体的な手順・使用ツール ③必要な前提条件（費用、スキル、環境） ④研究室で応用できそうな場面の示唆 | 「how-to」の骨子を箇条書きで抽出するよう指示し、実践可能性フラグの根拠にする |
| 専門交差タグ判定 | 防災シミュレーション/ABM/GISとの関連性の有無・関連度 | 別プロンプト（分類専用）でYes/No＋関連度スコアを出力させる |

### 5.3 出力フォーマット

LLMの出力はJSON形式（例: `{summary, category[], tags[], practicality_score, cross_domain_tags[]}`）に固定し、後続処理でパースしやすくする。JSON形式を強制するため、可能であればFunction Calling/Structured Outputを利用する。

### 5.4 検証・フォールバック

- LLM出力のJSONパースに失敗した場合は再試行（最大N回）、それでも失敗した場合は要約なしで「未要約」として一覧に残す。
- 要約文中に元記事本文に存在しない固有名詞・数値が含まれるかの簡易チェックは将来検討（v1では実施しない、10章の未確定事項とする）。

## 6. データ設計

### 6.1 記事テーブル（articles）カラム案

| カラム名 | 型 | 説明 |
|---|---|---|
| id | UUID/連番 | 主キー |
| source_id | FK | 情報源マスタへの外部キー |
| title | text | 記事タイトル |
| url | text | 元記事URL（重複判定にも使用） |
| published_at | datetime | 元記事の公開日時 |
| fetched_at | datetime | 収集した日時 |
| raw_content | text | 取得した本文/概要（要約前の元テキスト） |
| summary | text | LLM生成要約 |
| track_type | enum | `trend`（動向系）/ `practice`（実践系） |
| categories | text[] / JSON | カテゴリ分類（複数可、7カテゴリから） |
| tags | text[] / JSON | 自由記述タグ |
| cross_domain_tags | text[] / JSON | 専門交差タグ（防災/ABM/GIS等） |
| practicality_score | int/float | 実践可能性スコア（実践系のみ、例: 0-5） |
| is_duplicate | boolean | 重複記事フラグ |
| llm_process_status | enum | `pending`/`done`/`failed` |
| created_at / updated_at | datetime | レコード管理用 |

### 6.2 情報源マスタ（sources）カラム案

| カラム名 | 型 | 説明 |
|---|---|---|
| id | UUID/連番 | 主キー |
| name | text | ソース名 |
| url | text | RSS/APIエンドポイント |
| track_type | enum | 既定の系統（`trend`/`practice`） |
| fetch_method | enum | `rss`/`api` |
| is_active | boolean | 収集対象とするか |
| last_fetched_at | datetime | 最終収集日時 |

### 6.3 カテゴリ設計（v1確定案）

たたき台をベースに以下を正式カテゴリとする（複数カテゴリ付与を許可）。

1. 研究・新モデル動向（動向系）
2. ビジネス・企業動向（動向系）
3. 社会・政策・研究倫理（動向系）
4. 実践Tips・ツール活用（実践系）
5. メイカー系事例（ハード×AI）（実践系）
6. 生成系事例（コンテンツ×AI）（実践系）
7. 防災・ABM・GIS関連（横断タグ、上記1〜6のいずれとも併用可）

## 7. 非機能要件

| 項目 | 要件（案） |
|---|---|
| 実行頻度 | バッチは1日1回（毎朝、利用開始前に完了する時間帯、例: 6:00-7:00）を基本とする。手動再実行も可能とする |
| 可用性 | 個人利用のため高可用性は不要。日次バッチが多少遅延・失敗しても翌日リカバリ可能な設計でよい |
| コスト | 個人負担を想定し、LLM API利用量・サーバー費用ともに低コスト構成を優先する（月額目安は10章で検討） |
| 保存期間 | 記事データは当面無期限保存（個人の技術トレンド振り返りに使う可能性があるため）。ストレージ増大が問題になった場合に古い記事のアーカイブ/削除を検討 |
| パフォーマンス | 一覧表示は数百〜数千件規模を想定し、体感1〜2秒程度で表示できること |
| セキュリティ | 認証不要の前提だが、外部公開する場合は最低限のBASIC認証やIP制限を検討する |
| 保守性 | 情報源の追加・停止が容易に行えること（設定ファイルまたは管理UI経由） |
| 障害時対応 | バッチ失敗時にログで原因を確認できること。通知機能はv1では不要（自分で確認する運用） |

## 8. 画面構成

### 8.1 一覧画面（メイン画面）

- 画面上部: 系統タブ（「すべて」「動向系」「実践系」）
- 左サイドバー（またはドロップダウン）: カテゴリチェックボックスフィルタ（7カテゴリ）、横断タグフィルタ（防災/ABM/GIS）
- メインエリア: 記事カードを新着順にリスト表示
  - カード内容: タイトル、要約（2〜3行）、カテゴリバッジ、実践可能性スコア（実践系のみ星印等で表示）、公開日、ソース名
- ページング or 無限スクロール（件数次第）

### 8.2 詳細画面（モーダル or 別ページ）

- タイトル、全要約文、カテゴリ・タグ一覧、実践可能性スコアとその根拠（LLMが抽出したhow-to箇条書き）、元記事へのリンクボタン、公開日・収集日

### 8.3 情報源管理画面（簡易）

- 情報源一覧（名前、URL、系統、有効/無効切り替え）
- 追加・編集・削除フォーム
- 最終収集日時、直近バッチのステータス表示

### 8.4 バッチステータス確認（簡易ダッシュボードでも可）

- 直近実行日時、成功/失敗、新規収集件数、要約失敗件数

## 9. 技術構成案

個人開発・単独運用を前提に、複数の構成案を提示する。

### 案A: オールインワン軽量構成（推奨・v1向け）

- フロントエンド: Next.js（App Router）でSSR/SSGしつつ一覧・詳細を表示
- バックエンド: Next.jsのAPI Routes / Server Actionsで完結
- DB: SQLite（個人利用規模なら十分）またはSupabase(PostgreSQL)の無料枠
- バッチ実行: Vercel Cron Jobs（Next.jsをVercelにホスティングする場合）またはGitHub Actionsのscheduled workflow
- LLM: Claude API / OpenAI APIをバッチ処理内から呼び出し

メリット: 単一リポジトリ・単一言語（TypeScript）で完結し、学習コストと運用コストが低い。ホスティングも無料〜低コストで開始できる。
デメリット: SQLiteはVercelのようなサーバーレス環境では永続化に工夫が必要（外部DBが実質必須になる場合がある）。

### 案B: Python中心構成

- フロントエンド: Streamlit または FastAPI+簡易HTML/HTMX
- バックエンド/バッチ: Python（feedparser等でRSS収集、LLM呼び出しも同一言語）
- DB: SQLiteまたはPostgreSQL
- バッチ実行: cron（自宅サーバー/VPS）またはGitHub Actions

メリット: データ処理・LLM呼び出しまわりのエコシステムが豊富（研究者はPythonに慣れていることが多い）。
デメリット: Streamlit UIはカスタマイズ性がNext.jsより劣る。常時起動サーバーが必要な場合、VPS費用が発生する。

### 案C: サーバーレス+マネージドDB構成

- フロントエンド: Next.js もしくは Astro（静的寄り）
- バックエンド: Cloudflare Workers / Vercel Functions
- DB: Supabase もしくは Turso（SQLite互換のエッジDB）
- バッチ実行: Cloudflare Cron Triggers / Supabase Edge Functions + pg_cron

メリット: 常時起動サーバー不要で運用コストを最小化できる。スケールを気にする必要がほぼない。
デメリット: サーバーレスの実行時間制限（記事収集・LLM要約が多いと分割実行が必要になる場合がある）。構成要素が増えるため初期セットアップがやや複雑。

### 9.1 比較まとめ

| 観点 | 案A（Next.js一体型） | 案B（Python中心） | 案C（サーバーレス+マネージドDB） |
|---|---|---|---|
| 実装コスト | 低〜中 | 低（Python得意なら） | 中 |
| 運用コスト | 低 | 中（VPSの場合） | 低 |
| 拡張性 | 中 | 中 | 高 |
| 学習コスト | フロント/バック統一で低い | データ処理は楽、UI別途学習 | サーバーレス特有の制約理解が必要 |

**v1推奨**: 案A（Next.js一体型、DBはSupabase無料枠）。将来的にバッチ処理が重くなればPython製の収集・要約バッチのみ切り出す拡張も可能。
